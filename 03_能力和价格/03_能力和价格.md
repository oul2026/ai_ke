价格口径：API 标准实时调用价，单位均为“每 100 万 token”。表格固定按“输入、输出、缓存、思考”展示：

| 字段 | 含义 |
| --- | --- |
| 输入 | prompt、历史对话、上传文本、工具结果进入模型的 token |
| 输出 | 最终答案、函数调用、可见回复等普通输出 token |
| 缓存 | prompt cache / context cache 命中或写入价格 |
| 思考 | thinking / reasoning token 单价；多数厂商把它按输出 token 计费 |

所以“输出”和“思考”单价经常相同，但账单不同：不开思考只付可见输出；开思考要付“可见输出 + 隐藏思考”。如果 100 万可见输出同时生成 100 万思考 token，则输出侧成本约等于 `2 × 输出单价`。实际思考 token 数量只能从 API usage 看，不能按模型名猜。

## 成本与预算

```text
单次费用 = 输入 token × 输入单价
       + 可见输出 token × 输出单价
       + 思考 token × 思考单价
       + 缓存写入/读取费用
       + 搜索、代码执行、工具调用、重试等额外费用
```

预算时先估这几项：

| 项目 | 要看什么 |
| --- | --- |
| 单次输入 | 系统提示、用户问题、历史上下文、RAG 文档、工具结果 |
| 单次输出 | 最终回答长度、代码长度、JSON/表格体积 |
| 思考 token | reasoning effort、thinking budget、max_output_tokens |
| 调用量 | 每日请求数、并发、失败重试、Agent 多步循环 |
| 折扣 | 缓存、Batch/Flex、区域部署、企业合同 |

## 选型规则

1. 高风险任务先看能力，再看价格：合同、法律、医疗、代码架构、客户数据不能只按便宜选。
2. 高频低风险任务先看价格和吞吐：分类、抽取、格式转换、短文本改写可以用轻量档。
3. 思考模型必须看实际 usage：thinking token 可能比最终答案更贵。
4. 长上下文要单独实测：价格、延迟、缓存命中、工具结果膨胀都会改变账单。
5. 最终用同一组真实任务测试质量、速度、成本、失败率，不用单一排行榜决定采购。
